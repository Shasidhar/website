Building Spark Streaming App - Part 5 (Kafka Integration)

08 Jul 2016

In previous blog we discussed about how we integrate zeppelin to our streaming processing engine for visualizing the results. You may be thinking we already have the complete working streaming application, why we need to develop more for the same application. From this blog its more of productionizing the application, where we solve some of the critical pieces in the system.

Why to use Kafka?

One of the major drawback of our application is, we are reading the data through filestreams, its always an issue if we use files as our input source for streaming application. One of the major drawback of file stream is, we cannot track how much of the data is processed by the streaming application in case of application crash. In order to be able to know how much data is exactly being processed in the streaming application we need an intelligent streaming data source where we should be able to track the progress.

In order to overcome the issues which I explained above we need some intelligent input data source for our system. I think Kafka is one of the most popular streaming data source available today. So what exactly is kafka? Apache kafka is a publish subscribe distributing messaging system.

Kafka as an input source

In spark streaming we have a very good support to use kafka as in input source for streaming application. Spark has a built in support for kafka receiver, where we can use kafkaâ€™s topic as the source of streaming data. There are two different ways we can use kafka as a source,

  1. Receiver based approach
  2. Direct approach

In receiver based approach we use the traditional spark receivers to receive the data and process in spark streaming. Where as in direct approach we dont create spark receivers, but we pull the data periodically directly from kafka and process in spark side. With direct approach we get more advantages like Simplified paralellism, Efficiency and Exactly once semantics. For more details about the difference between these two approaches refer Spark Streaming + Kafka Integration Guide

Kafka direct receiver

In our application now its obvious to use kafka direct approach. To create kafka direct stream in spark there is a direct API.


	KafkaUtils.createDirectStream[keytype, valuetype, keydecoder, valuedecoder](
    	    streamingContext, kafkaParams, topicsSet)

There are different parameters we need to pass in the above code snippet.

  1. keytype - Defines what is the type of key in Kafka topic
  2. valuetype - Defines what is the type of value in Kafka topic
  3. keydecoder - Controls how the key is decoded in spark side.
  4. valuedecoder - Controls how the value is decoded in spark side.
  5. streamingContext - It is the spark streaming context
  6. kafkaParams - Map of additional properties to connect with kafka. In our case we are passing brokerids
  7. topicsSet - Set of topics from which we want to create input stream.

The return type is a DStream[Key,Value] pair, where both key and value is of String type.

In our earlier receiver, that is file stream we were creating only the DStream[String]. Where String will be the actual content we want. In the new receiver we wanted only DStream[String] instead of DStream[Key,Value]. So in this case we can use map operation and project only the value out of these pair to get DStream[String].

Once we have the input DStream[String] we can use all other old code which we have for calculating the results and aggregates.

For complete code refer kafka branch of Sensor analytics repository

This completes one kafka integration with the spark streaming application. If you observe the code change we made to integrate with kafka , its only 4 or 5 lines of code change. This shows how much flexibile we can be with spark and work with different parts of the application in modular way. So, next step in the series would be to integrate Cassandra into the system.
